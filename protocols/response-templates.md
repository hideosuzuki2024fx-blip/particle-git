# Response Templates (Operational)

目的: 失敗時・不確実時に、捏造せず、報告粒度を揃え、運用を止めないための定型文テンプレ集。

## 1) URL 参照（web.run）成功時
- 取得できた範囲のみを要約し、引用は短く（必要最小限）にする
- 参照元（URL）を明記する

テンプレ:
- 「URL（<URL>）にアクセスを試み、<取得できた範囲> を取得しました。取得した範囲に基づく結論は <結論> です。」
- 「この結論は <根拠となる箇所> に基づきます。」

## 2) URL 参照（web.run）失敗時
- 本文推測はしない
- 「URL・失敗段階・エラー内容のみ」を報告する

テンプレ:
- 「<URL> へのアクセスを試みましたが、<段階: 例) open/click/fetch> の時点で失敗しました。エラー: <エラー内容>。本文は取得できていないため、この情報からは判断できません。」

## 3) web ツール未実行（または利用不可）
テンプレ:
- 「この環境からはそのURLの内容は分かりません。」

## 4) Git リポジトリ安全チェック失敗（PowerShell）
- git ルート特定/inside-work-tree 判定に失敗したら、以降のファイル操作・git操作を停止する

テンプレ（ユーザー向け説明）:
- 「Git作業ツリーであることを確認できませんでした（git チェック失敗）。想定外の場所でのファイル操作を避けるため、以降の処理は中止します。」
- 「正しいリポジトリパス/カレントディレクトリを提示してください。」

## 5) 長文コード（チャットで分断リスク）
- コード全文をチャットに展開せず、生成すべきコードをファイルとして作るスクリプトを提示する
- そのスクリプトはファイル作成→git add→git commit まで自己完結

テンプレ:
- 「コードが長く、1つのコードブロックとして安全に表示しきれない可能性があるため、コード本文を `./<path>/<name>` として生成し、コミットまで行う PowerShell スクリプトを提示します。」

## 6) 不明点が残る（情報不足）
- 推測で埋めない
- 追加で必要な情報を「列挙」する（質問攻めにせず最小限）

テンプレ:
- 「この情報からは判断できません。判断に必要なのは次の<最小個数>点です: <項目1>, <項目2>。」
